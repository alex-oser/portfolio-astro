A few years ago I wrote about [building this site with Gatsby](/projects/portfolio). It served me well, but somewhere along the way the joy drained out of it. Dependency upgrades turned into afternoons, the `gatsby develop` cache needed regular sacrifices to keep working, and every time I sat down to actually _write_ something I first had to remember how the GraphQL layer was wired together. When the tool you built to lower the friction of writing becomes the friction, it is time for a change.

So this month's project was migrating the whole thing to [Astro](https://astro.build). I wanted to keep everything I liked — MDX posts, my MUI components, the same URLs — and throw away everything I didn't. Here is how it went.

# Why Astro

The pitch that sold me is that Astro ships zero JavaScript by default and only hydrates the interactive bits you ask it to. Most of my site is just content. There is no reason to send React down the wire to render a paragraph about a side project. But I _did_ still want React and MUI for the parts that are genuinely interactive, and Astro's "islands" let me have both. Static where it can be, interactive where it needs to be.

# Content collections instead of GraphQL

In Gatsby, my `projects`, `snippets`, and `blog` posts were sourced from the filesystem and queried through GraphQL. It worked, but it always felt like a lot of ceremony to ask "what posts do I have?".

Astro's content collections replace all of that with a typed schema and a plain function call. I describe the shape of my frontmatter once in `src/content.config.ts`:

```ts
const projects = defineCollection({
  loader: glob({ base: './src/content/projects', pattern: '**/*.{md,mdx}' }),
  schema: z.object({
    title: z.string(),
    status: z.enum(['live', 'in progress', 'idea', 'dead']),
    caption: z.string(),
    link: z.string().url().optional(),
    repo: z.string().url().optional(),
    date: z.coerce.date(),
  }),
});
```

Now if I typo a `status` or forget a `date`, the build tells me before I ever open a browser. And fetching posts is just `getCollection('projects')` — no query language required.

# Pages from files, not config

The most Gatsby-specific code I had was `gatsby-node.js`, where I programmatically created a page for every post with `createPages`. In Astro that whole file disappears. A single dynamic route does the same job:

```astro
---
export async function getStaticPaths() {
  const projects = await getCollection('projects');
  return projects.map((entry) => ({
    params: { slug: toSlug(entry.data.title) },
    props: { entry },
  }));
}

const { entry } = Astro.props;
const { Content } = await render(entry);
---

<PostLayout title={entry.data.title} caption={entry.data.caption}>
  <Content />
</PostLayout>
```

I derive the slug from the title exactly like I used to, so every old URL still works. That mattered to me — migrations that quietly break your links are not migrations, they are demolitions.

# Code blocks that ship no JavaScript

This is my favorite part. In the old site I had a `<CodeRenderer>` component built on `prism-react-renderer`, and because MDX v1 was fussy about it, I stored my code samples as escaped strings in little `code.ts` files and imported them in. It worked, but it was deeply silly, and shipping a syntax highlighter to the client to render code that never changes always bothered me.

Astro highlights code at _build time_ with [Shiki](https://shiki.style/). I deleted every `code.ts`, dropped the component, and went back to plain fenced code blocks:

````md
```ts
const mac = () => spoofYourMacAddress();
```
````

One line of config picks the theme:

```js
markdown: {
  shikiConfig: { theme: 'night-owl' },
},
```

The result is the same `nightOwl` look I always had, except now it is pure HTML and CSS. Zero kilobytes of JavaScript.

# Keeping the UI as islands

I had no interest in rewriting all my MUI components, so I didn't. They came over almost verbatim — `styled` and `sx` are unchanged — and now live as React islands. The trick is that each island that uses MUI needs the `ThemeProvider` in its own tree, so I wrapped my entry points in a small `Providers` component and mounted them with a hydration directive:

```astro
<SiteHeader client:idle siteTitle={SITE_TITLE} />
<Home client:load projects={projects} snippets={snippets} />
```

The header smooth-scrolls, so it hydrates. The static prose around it does not. Exactly the split I wanted.

Co-located images came along for free too. Where Gatsby needed `gatsby-plugin-image` and `gatsby-remark-images`, Astro just optimizes any image I reference relatively from an MDX file. The `StaticImage` imports turned into ordinary `![alt](./thing.png)` and got smaller `webp` files out the other side.

# One gotcha worth your time

If you use [Astro's local fonts](https://docs.astro.build/en/guides/fonts/) _and_ a CSS-in-JS library like MUI, here is the half hour I will save you.

Astro does not register an `@font-face` named `Raleway`. It generates a hashed family name behind the scenes and only exposes it through a CSS variable like `--font-raleway`. My global stylesheet used the variable, so my prose looked right — but my MUI theme said `fontFamily: "Raleway, Roboto"`, a literal name that matched nothing, so every island silently fell back to a serif. The fix is to point the theme at the variable instead:

```ts
typography: {
  fontFamily: "var(--font-raleway), Roboto, sans-serif",
},
```

The rule of thumb: reference Astro fonts by their `--font-*` variable everywhere, never by the human-readable name.

# Was it worth it?

Yes, easily. `@astrojs/rss` rebuilt my feed at the same `/rss.xml`, the sitemap comes from an integration, and the dev server starts instantly. But the real win is quieter than any benchmark: the project I had to relearn every time I opened it now mostly stays out of my way. Which means I am more likely to actually write the next post — like this one 😀
